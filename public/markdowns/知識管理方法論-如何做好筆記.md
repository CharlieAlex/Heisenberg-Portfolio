## 卡片盒筆記法 (Zettelkasten)

- [風靡歐美的卡片盒筆記法是什麼？12個法則和重點整理 | 閱讀前哨站](https://readingoutpost.com/smart-notes/)
- [zettelkasten卡片盒筆記法-建立知識連結網路來活用筆記](https://medium.com/pm%E7%9A%84%E7%94%9F%E7%94%A2%E5%8A%9B%E5%B7%A5%E5%85%B7%E7%AE%B1/zettelkasten%E5%8D%A1%E7%89%87%E7%9B%92%E7%AD%86%E8%A8%98%E6%B3%95-%E5%BB%BA%E7%AB%8B%E7%9F%A5%E8%AD%98%E9%80%A3%E7%B5%90%E7%B6%B2%E8%B7%AF%E4%BE%86%E6%B4%BB%E7%94%A8%E7%AD%86%E8%A8%98-f85a91729521)

### 1. 創始人與起源

- **創始人**：由 20 世紀德國社會學家 **尼克拉斯·盧曼 (Niklas Luhmann)** 系統化。
- **為了解決什麼問題**：
    - **資訊遺忘**：傳統筆記分散在各處，如果不建立連結，時間久了就會被遺忘且不具價值。
    - **線性思考的侷限**：傳統筆記本是按時間排列的線性架構，難以處理複雜、非線性、跨領域的知識。
    - **寫作困境**：解決「面對空白頁面不知從何寫起」的問題。盧曼透過卡片間的「對話」來自動產出論文內容。

### 2. 具體方法論 (筆記的三層次)

#### 1. 靈感筆記 (Fleeting Notes)

- **目的**：捕捉稍縱即逝的念頭。
- **案例**：在公車上想到：「複利效應是不是也能應用在人際關係的信任積累？」
- **特點**：雜亂、簡短，通常 24 小時內會被處理或刪除。

#### 2. 文獻筆記 (Literature Notes)

- **目的**：確保你「真的讀懂了」。
- **案例**：讀完《原子習慣》，用自己的話寫下：「作者認為習慣的養成不在於目標，而在於系統。系統是由微小的原子行為構成的。」
- **特點**：附上來源標註，**嚴禁直接複製貼上**。

#### 3. 永久筆記 (Permanent Notes)

- **目的**：提煉出「原子化」且「可重複利用」的觀點。
- **案例**：標題為 `[[系統思維優於目標導向]]`。內容寫到：「目標是短期的，一旦達成就會失去動力；而系統是長期的，它能透過持續的微調產生複利（連結回人際關係信任筆記）。」
- **特點**：一個檔案只講一件事，且能獨立存在。

#### 4. MOC (Map of content)

- 針對一個概念將筆記之間的關係串聯出來。

###  實作建議與注意事項

根據你的觀察與研究，整理出以下實踐指南：

#### 1. 連結的必要性

- **法則**：每一則文獻筆記與永久筆記都應被連結超過一次以上。
- **原因**：如果一則筆記無法與現有筆記產生連結，它就不具備發揮價值的潛力。

#### 2. 原子化的靈活處理 (避免死板)

- 並非所有單一概念都適合整理成永久筆記。
- **案例：Solow Growth Model**。如果硬要將模型中的每一個假設拆成原子卡片，會導致模型失去其結構價值。
- **解法**：接受具備脈絡的「模型卡片」，並利用視覺化工具（如 Canvas 或 Map）來呈現這些複雜的架構。

#### 3. 格式不決定本質

- 不要被 Markdown 或 Canvas 這種檔案格式限制了筆記的性質。
- 視覺化的 Canvas 同樣可以用來撰寫文獻筆記，它只是讓我們更清楚地理解部分與整體的層次關係。

#### 4. 避免「收藏家謬誤」

- 不要只是大量搬運文獻（Copy-paste），那是「文獻筆記」未完成的狀態。
- 真正的價值在於將文獻轉化為你自己的「永久筆記」，即你對事實的主觀解讀。

## 長青筆記 (Evergreen Notes)

[優化 Zettelkasten 卡片盒筆記法，試試Evergreen Note (長青筆記)](https://ithelp.ithome.com.tw/m/articles/10273883)

### 1. 創始人與起源

- **創始人**：由當代知識管理研究者 **Andy Matuschak** 提出。
- **為了解決什麼問題**：
    - **資訊過載而認知淺薄**：現代人讀了很多，但很少轉化成自己的思考。
    - **筆記的停滯性**：解決筆記寫完就再也不動的問題，提倡筆記應該像植物一樣不斷生長與迭代。

### 2. 具體方法論

長青筆記在卡片盒的基礎上，更強調「以概念為中心」的寫作流程：

- **原子化 (Atomic)**：一則筆記只處理一個核心概念。
- **以概念為導向 (Concept-oriented)**：筆記應以「概念」命名，而非以「書籍名稱」或「專案」命名。
- **密集的連結 (Densely linked)**：筆記之間必須有豐富的雙向連結，這也是其核心價值來源。
- **持續迭代**：筆記不是寫完就結束，而是會隨著你對主題理解的加深而不斷重寫。

## PARA + CODE

- [如何分類筆記？一套簡單又通用的分類架構 — PARA](https://medium.com/pm%E7%9A%84%E7%94%9F%E7%94%A2%E5%8A%9B%E5%B7%A5%E5%85%B7%E7%AE%B1/%E5%A6%82%E4%BD%95%E5%88%86%E9%A1%9E%E7%AD%86%E8%A8%98-e25c4cc39dba)
- [為什麼我不用 PARA 了？](https://raymondhouch.com/notion/para-pai/)

| **階段 (CODE)**     | **對應動作**        | **PARA 儲存位置**          | **筆記種類對應**      |
| ----------------- | --------------- | ---------------------- | --------------- |
| **Capture (獲取)**  | 隨手記下念頭、存下網頁     | **Daily Note / Inbox** | 靈感筆記            |
| **Organize (組織)** | 決定這則資訊對哪個目標有用   | **P / A / R**          | 分類與標籤管理         |
| **Distill (萃取)**  | 用自己的話重述，精煉出原子命題 | 保持在原處或移入 **Resources** | 文獻筆記 $\to$ 永久筆記 |
| **Express (表達)**  | 拼湊卡片，完成論文、專案或模型 | 從 **P / A / R** 提取內容   | MOC / 專案成果      |

### 創始人與起源

- **創始人**：由當代生產力專家 **Tiago Forte**（《打造第二大腦》作者）提出。
- **為了解決什麼問題**：
    - **資訊囤積症**：傳統按「主題」分類（如：經濟學、歷史、食譜）容易導致資料夾越來越深，最終找不到東西。
    - **無法轉化為行動**：筆記與實際要完成的工作脫節，導致在需要產出時，必須花費大量時間重新檢索。
    - **維護成本過高**：解決「分類強迫症」，讓人不再糾結某則筆記該放哪，而是快速決定它對目前的「行動」是否有幫助。

###  PARA 方法論 (四大分類)

PARA 根據**行動的緊急程度**將所有資訊分為四個區域：

1. **P (Projects) 專案**：
    
    - 定義：目前正在進行、且有明確「截止日期」與「具體目標」的短期任務。
    - 案例：撰寫《卡片盒筆記法》教學講義、規劃下週的旅行。
        
2. **A (Areas) 領域**：
    
    - 定義：需要長期維持一定水準的責任範圍，沒有明確截止日。
    - 案例：身體健康、財務管理、房東責任、持續更新的筆記系統。
        
3. **R (Resources) 資源**：
    
    - 定義：感興趣的主題、參考資料，或未來可能有用的資訊。這更像是個人的「數位百科全書」。
    - 案例：經濟學模型研究、黑膠唱片收藏筆記、CSS 排版技巧。
        
4. **A (Archives) 封存**：
    
    - 定義：已完成、已失敗或目前不再感興趣的以上三類內容。

### CODE 方法論 (四大分類)

CODE 代表了資訊從「進入系統」到「產生價值」的四個階段，它能與你提到的卡片盒筆記循環完美結合。

1. **C (Capture) 獲取：只保留對你有啟發的東西**

	- **定義**：捕捉外部資訊或內在想法的過程。
	- **工具對應**：對應到你的**靈感筆記 (Fleeting Notes)** 與 **Daily Note**。
	- **關鍵準則**：
	    - **策展心態**：不要像囤積者一樣什麼都存，只捕捉讓你產生共鳴、驚訝或有用的資訊。
	    - **外部化**：將大腦的負擔轉移到系統中，例如快速記下：「Solow Model 的儲蓄率假設似乎有盲點」。

2. **O (Organize) 組織：為了「行動」而分類**

	- **定義**：將捕捉到的資訊放入 **PARA** 架構中。
	- **核心邏輯**：
	    - **不要按主題分類，要按專案分類**。這能解決你提到的「資訊囤積症」，讓資訊隨時處於可被調用的狀態。
	    - 問自己：「這則筆記在哪個**專案 (Project)** 中最能幫到我？」如果不屬於任何專案，就放入**領域 (Area)** 或**資源 (Resource)**。

3. **D (Distill) 萃取：尋找核心本質**

	- **定義**：將長篇大論轉化為精華的過程。
	- **工具對應**：這正是將**文獻筆記 (Literature Notes)** 轉化為**永久筆記 (Permanent Notes)** 的關鍵步驟。
	- **技術實踐**：
	    - **漸進式總結 (Progressive Summarization)**：透過粗體、螢光筆標記，分層過濾資訊，直到剩下最精華的原子概念。
	    - **主觀解讀**：誠如你提到的，這步驟是加入「主觀建模」的過程，例如將耶穌故事萃取成「科技力」或「神蹟論」。

4. **E (Express) 表達：分享與產出**

	- **定義**：將積累的知識應用在實際產出上。
	- **核心邏輯**：
	    - **中間產物 (Intermediate Packets)**：不要試圖一次寫完大論文，而是利用平時累積的「原子化永久筆記」進行組裝。
	    - **解決寫作困境**：當你需要表達時，你不是面對空白頁面，而是從你的卡片盒（或是 PARA 的 Resources）中提取已萃取好的模塊進行拼接。

### 注意事項與實作建議

#### 1. 避免過度分類

- PARA 的目的不是為了精確，而是為了快速行動。
- 如果一個知識點目前沒有具體的專案可對應，就應該放在「資源」或「領域」，甚至僅作為一張獨立的卡片存在。

#### 2. 行動導向的連結

- 在實作上，你可以將 PARA 作為「資料夾」或「標籤」的結構，而將 Zettelkasten 作為「內容」的連結邏輯。
- 正如你提到的，永久筆記的核心是單一概念，是否滿足標題命題並不重要，重點是它在系統內的意義。

#### 3. 定期清理與封存

- PARA 的生命力在於「Archives」。不要害怕把沒用的東西移走。
- 這解決了 Zettelkasten 可能產生的「死筆記」問題：如果一則筆記在可預見的未來都無法被連結，應考慮其必要性。

#### 4. 格式與工具的結合

- 你可以利用 **Obsidian Canvas** 作為專案（Project）的視覺化控制台，將分散在各處的資源（Resources）連結進來。
- 文獻筆記可以直接以視覺化地圖形式呈現，將其組成部分放在 Canvas 裡理解。

## 子彈筆記 (Bullet Journal, BuJo)

[子彈筆記術(上) Bullet Journal 快狠準清單日記法教學](https://www.playpcesor.com/2015/12/bullet-journal.html?m=1)

### 創始人與起源

- **創始人**：由數位設計師 **Ryder Carroll** 提出。
- **為了解決什麼問題**：
    - **注意力分散**：創始人患有 ADHD（注意力不足過動症），需要一套低摩擦、能快速捕捉資訊的系統。
    - **數位疲勞**：現代 App 充滿干擾，人們需要重回「紙筆」的慢思考過程（雖然現在也有數位實踐）。
    - **資訊過載**：解決「什麼都想做」的焦慮，透過手動轉移（Migration）強迫使用者審視任務的必要性。

### 具體方法論 (核心組件)

子彈筆記的精髓在於 **快速記錄 (Rapid Logging)** 與 **符號系統**：

1. **符號系統 (Key)**：
    
    - `•` 任務 (Task)
    - `X` 已完成 (Task Complete)
    - `>` 遷移 (Migrated)：將任務移至隔天或下個月。
    - `O` 事件 (Event)
    - `–` 筆記 (Note)：純粹的資訊記錄。
        
2. **核心模組 (Collections)**：
    
    - **索引 (Index)**：位於筆記本開頭，記錄各內容頁碼。
    - **未來規劃表 (Future Log)**：記錄未來幾個月的大事。
    - **月計畫 (Monthly Log)**：當月的概覽與任務清單。
    - **日計畫 (Daily Log)**：每天的快速捕捉。這對應你提到的 **靈感筆記 (Fleeting Note)** 空間，用來捕捉當天想到的任何東西。

## 深度專題探討

### 主題一：從「線性產出」到「互補循環」

筆記不是 $A \to B \to C$ 的單向過程。相反，它們是一個**生態系統**：

- **靈感**驅動**文獻**研究。
- **永久筆記**的反思會催生新的**靈感**。
- **教學建議**：不要強迫自己一定要從靈感開始。有時是先看了一篇論文（文獻），才回頭去補捉靈感。

### 主題二：格式 $\neq$ 種類（視覺化的本質）

Canvas（畫布）或 Markdown（純文字）僅是「格式」。

- **核心觀點**：如果一個經濟學模型（如 Solow Model）本身具備強烈的階層性，那麼一份長文 Markdown 可能比散落的卡片更合適。
- **應用**：在 Obsidian 中，你可以把 Canvas 當作「文獻的拆解地」**，等邏輯理清了，再把精華收納進永久筆記。

### 主題三：原子化的陷阱（以經濟研究為例）

**並非所有概念都適合「原子化」**。

- **過度拆解的風險**：強行拆解 Solow Model 會導致脈絡丟失。
- **解法**：接受「中觀筆記」的存在。有些卡片本身就是一個「模型卡」，它連結了多個前提，但它作為一個整體被引用。

### 主題四：Heptabase 的「預設永久」邏輯

Heptabase 傾向於讓每張卡片都存在於 Card Library。

- **給 Obsidian 使用者的建議**：不要害怕在 `[[ ]]` 創建新頁面。即使那張卡片現在看起來很廢，只要它有了標題與連結，它就在你的知識網路中佔了一席之地。

## 實戰：如何使用 obsidian 實行這些筆記方法

### 一、 捕捉與靈感：Daily Note 作為靈感入口

- **工具對應**：**Daily Note (日誌)**。
- **實作邏輯**：Daily Note 是一個低摩擦的捕捉空間，非常適合承載「靈感筆記（Fleeting Notes）」。
- **操作建議**：將每天產生的零碎念頭、待辦事項或初步觀察記錄在當日的日誌中。這能避免在思考初期就陷入「該把這張卡片放在哪」的分類焦慮，確保大腦的注意力集中在捕捉內容本身。

### 二、 知識網絡：Backlink 與 Graph View 發展永久筆記

- **工具對應**：**反向連結 (Backlinks)** 與 **關係圖譜 (Graph View)**。
- **實作邏輯**：
    - **雙向連結 (`[[ ]]`)** 取代了傳統卡片盒複雜的編號系統，讓筆記之間建立直接聯繫。
    - **Graph View** 則視覺化了筆記間的連結密度與群聚現象。透過觀察圖譜中哪些節點特別活躍，你可以發現潛在的知識聯繫，進而將分散的靈感與文獻提煉成更具主觀見解的「永久筆記」。

### 三、 關係脈絡：Canvas 圖像化與 MOC 構建

- **工具對應**：**Canvas (畫布)**。
- **實作邏輯**：
    - Canvas 不僅是畫圖工具，它能將文獻筆記的各個部分以視覺化的位置關係組合在一起，具體展現概念間的脈絡。
    - 在處理如 **Solow Growth Model** 等複雜模型時，Canvas 能在不強迫拆解的情況下，保留模型各組成部分（如假設、推導）的階層性架構。這就是實踐 **MOC (Map of Content)** 最直觀的方式。

### 四、 概念提煉：Note Refactor 實現原子化

- **工具對應**：**Note Refactor 插件**。
- **實作邏輯**：長青筆記（Evergreen Notes）強調筆記應具備「原子化」與「以概念導向」的特性。
- **操作建議**：當你在寫一份長文獻筆記時，若發現其中某段文字具備獨立發展的價值，可以使用 Note Refactor 快速將其提取成獨立的永久筆記。這能確保每則筆記只處理一個核心概念，方便未來跨主題的重複引用。

### 五、 生活管理：Markdown 格式與 Task Status 實現 BuJo

- **工具對應**：**Markdown 清單** 與 **Tasks/Custom Task Status 插件**。
- **實作邏輯**：
    - 利用 Markdown 的 `*` 或 `-` 列表，搭配自定義符號（如 `[>]` 代表遷移、`[/]` 代表進行中），可以完美復刻子彈筆記的符號系統。
    - **Task Status 插件** 能讓你透過快捷鍵切換任務狀態，並在 Daily Note 中快速捕捉生活瑣事與靈魂碎片。
- 舉例說明
	- [ ] Todo: Easy or Fast
	- [x] Done
	- [/] In Progress
	- [-] Cancel
	- [I] Idea: maybe Todo, maybe Cancel
	- [f] Urgent Todo
	- [<] Scheduling Todo
	- [>] Archived
